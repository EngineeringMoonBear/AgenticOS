# Dev/GitHub PR-Triage Connector — Design

**Date:** 2026-06-08
**Status:** Approved (brainstorm) — pending implementation plan
**Author:** Josh + Claude
**Scope:** The **first** AgenticOS domain connector. Each connector is its own
brainstorm; this spec covers only the dev/GitHub PR-triage connector.

---

## 1. Goal

Give Hermes (the agent brain) its first real-world reach: a daily, read-only
scan of open pull requests across the GitHub org that produces a "what needs
your attention" digest in the vault and a record in the task ledger. It turns
the platform from purely *observing itself* into *observing your dev domain*.

## 2. Why this connector first

- **Reachable today** — GitHub REST is available from the Droplet with a token;
  no new infra (unlike smart-home/networking, which need a LAN-resident Pi).
- **Highest dogfood value** — Josh lives in these repos daily.
- **Lowest stakes to get the pattern right** — read-only, no business/farm risk,
  before tackling higher-stakes domains.
- **Establishes the connector pattern** — proves the five-step shape
  (fetch → assess → render → write → record) the future "connector interface"
  will extract.

## 3. Architecture

A new Hermes **cron task** `pr_triage`, registered like `daily-brief` /
`vault-ingest` and run by the `hermes-gateway` scheduler. No new container, no
public webhook ingress. Each run executes five steps:

```
cron tick (07:30 local)
  → fetch    GitHub REST: org repos → those with open PRs → per-PR facts
  → assess   pure function: classify each PR into buckets
  → render   deterministic markdown digest + short local-SLM prose intro
  → write    overwrite living vault note /opt/vault/wiki/_meta/dev-pr-digest.md
  → record   insert a Postgres `tasks` row (kind='pr-triage') with bucket counts
```

Downstream, for free:

- `vault-ingest` (hourly) picks the note into OpenViking → feeds agent memory.
- The dashboard **Runs** view shows the `pr-triage` task (reads Postgres).
- The dashboard **Memory** tab can open the digest note (reads the vault).

No new dashboard code is required for the MVP.

### 3.1 Composition principle

This mirrors the existing `inbox-watcher → inbox_triage` and
`vault-ingest` / `daily-brief` patterns. A "connector" = **ingest** (cron poll
or daemon) + **process** (a task that records to Postgres and writes to the
vault, optionally routing to a worker) + **surface** (the dashboard already
reads Postgres + vault). PR-triage is the cron-poll variant.

## 4. Components

Each unit is small, single-purpose, and independently testable.

### 4.1 `tasks/pr_triage.py` — entrypoint/orchestrator

- Orchestrates fetch → assess → render → write → record.
- Reuses the same `record_task_start` / `record_task_completion` shape as the
  other tasks in `agenticos_hermes.tasks` (if a shared db helper already exists,
  use it; otherwise follow the per-module pattern already in the codebase).
- Records `kind='pr-triage'`; metadata carries per-bucket counts + any per-repo
  errors; status `done` unless the whole run failed (e.g. no token).
- Invoked by `python -m agenticos_hermes.tasks.pr_triage` (cron + manual).

### 4.2 GitHub client — `workers/github_client.py` (or a focused module)

- Talks to the GitHub REST API via `httpx` with a bearer token.
- Functions:
  - `list_org_repos(org)` → repos (paginated), filter to non-archived.
  - `list_open_prs(repo)` → open PRs (paginated; skip repos with none).
  - `pr_facts(repo, pr)` → `{number, title, author, draft, updated_at,
    mergeable_state, review_decision, checks_state}` where `checks_state` is the
    rolled-up CI conclusion (success/failure/pending) from the PR head SHA's
    check-runs/commit-status.
- Read-only. No write calls anywhere in v1.
- Pagination + a small per-call timeout; surfaces HTTP errors to the caller.

### 4.3 Assessment — pure function

- `assess_pr(facts, now, stale_days) -> set[str]` (or an ordered list) assigning
  buckets:
  - `ci-failing` — head checks concluded failure.
  - `needs-review` — open, non-draft, no approving review decision yet.
  - `ready-to-merge` — approved + CI success + mergeable.
  - `has-conflicts` — `mergeable_state == 'dirty'`.
  - `stale` — `updated_at` older than `stale_days` (default 7).
  - `draft` — draft PR (informational; excluded from "needs attention").
- Pure, deterministic, no I/O → the primary unit-test target.

### 4.4 Renderer — markdown digest

- `render_digest(assessed, generated_at) -> str`:
  - A "🔔 Needs your attention" section (ci-failing, needs-review,
    has-conflicts, ready-to-merge) ordered by priority.
  - A per-repo table (repo · PR · author · buckets · age).
  - A short prose intro generated by the **local SLM** (Ollama via the existing
    `workers.slm_runner`) summarizing the headline counts. The deterministic
    body is authoritative; the SLM only writes the intro. If the SLM call fails,
    fall back to a one-line deterministic intro (no hard failure).

### 4.5 Output writer

- Overwrite a **living** note at `/opt/vault/wiki/_meta/dev-pr-digest.md` (a
  current snapshot, like `curator-log.md`), with a front-matter `generated_at`.
- The Postgres `tasks` row provides the run history; the note is always "now".
- Write the file world-readable (`0644`) so the cap-restricted hermes container
  and vault-ingest can read it (the readability lesson from the soak).

## 5. Configuration & auth

- **Token:** a **fine-grained GitHub PAT**, read-only scopes: Contents (read),
  Pull requests (read), Checks (read), Metadata (read). Stored as `GITHUB_TOKEN`
  in `/opt/agenticos/.env`. **Josh provisions it** (creating tokens is the
  user's responsibility; the assistant does not create credentials). Compose
  injects it into the `hermes-agent`/`hermes-gateway` container env.
- **Org:** `GITHUB_ORG` (default `EngineeringMoonBear`).
- **Tunables (env, with defaults):** `PR_TRIAGE_STALE_DAYS=7`,
  `PR_TRIAGE_VAULT_NOTE=/opt/vault/wiki/_meta/dev-pr-digest.md`.
- **Schedule:** cron `30 7 * * *` (07:30 local), registered idempotently in
  `infra/scripts/register-cron-jobs.sh` next to the existing jobs.

## 6. Cost

- Assessment + rendering body = pure code → free.
- Prose intro = local Ollama SLM → free (no Max/API spend).
- GitHub REST authenticated limit is 5000/hr — a daily poll of one org is well
  under it.
- **Net marginal cost: ~$0**, within the Max + DigitalOcean budget.

## 7. Error handling

- **Per-repo isolation:** a failure fetching one repo/PR does not abort the run;
  collect the error, continue, and include an `errors` list in the task
  metadata + a footer in the digest. (The "don't swallow errors" lesson from
  the vault-ingest fix.)
- **Missing/invalid token:** fail the run loud — status `failed`, a clear error
  in the task record; no partial digest written.
- **SLM failure:** degrade to a deterministic intro; the run still succeeds.
- **GitHub rate limit / 5xx:** surface in the per-repo error; the next daily run
  retries naturally.

## 8. Testing (TDD)

- **Assessment** — unit tests over fixture PR-fact payloads asserting bucket
  classification (each bucket + combinations + the stale boundary).
- **Renderer** — snapshot/segment test from fixed assessed input (stable output,
  SLM intro stubbed).
- **GitHub client** — mocked `httpx` (same approach as the `vault_ingest` `rm`
  test) asserting read-only calls, pagination, and fact extraction; never hits
  the network in tests.
- **Orchestrator** — DB-gated test (mirrors existing task tests) asserting a
  `pr-triage` row is recorded with the expected metadata shape.

## 9. Surfacing (MVP vs later)

- **MVP:** vault note (Memory tab) + Postgres `tasks` row (Runs view) — both
  already wired; no dashboard changes.
- **Later (out of scope):** a dedicated dashboard "Dev" panel/tab.

## 10. Out of scope (YAGNI)

- Writing/acting on GitHub (labels, comments, merges) — **read-only in v1**; a
  write capability is a separate, explicitly-gated follow-up.
- Real-time webhooks / event-driven daemon (Approach B) — unneeded for a daily
  digest.
- Multiple orgs / collaborator repos outside the org.
- Issue triage (a possible later dev-connector capability).
- Extracting the generic "connector plugin interface" — earned after connector
  #2 gives a real second case to factor against.

## 11. Acceptance criteria

- `python -m agenticos_hermes.tasks.pr_triage` run on the Droplet:
  - Produces `/opt/vault/wiki/_meta/dev-pr-digest.md` listing real open PRs
    across `EngineeringMoonBear`, with a "needs attention" section.
  - Records a `pr-triage` row in Postgres `tasks` with per-bucket counts.
  - Exits `done` with `errors: []` when the token is valid and all repos fetch.
- The cron job `pr-triage` is registered and visible in `hermes cron list`.
- A subsequent `vault-ingest` ingests the digest note into OpenViking.
- `soak-healthcheck.sh` can be extended later to assert the digest note is fresh
  (optional follow-up, not required for acceptance).
- All unit tests pass; the connector adds **$0** recurring cost.
